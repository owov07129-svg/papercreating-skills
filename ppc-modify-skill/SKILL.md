# PPC Modify Skill — 论文修改与版本管理

在 ppc-edit-skill 生成的论文基础上，按用户要求修改。**保留旧版、生成新版。**

## 前置条件

1. **排版大纲**已由 ppc-template-skill 生成并保存为文件（如 `排版大纲.md`）
2. **论文初稿**已由 ppc-edit-skill 生成（含 `main.tex` + `main.pdf` 等）
3. 本地工具链可用（必要时先跑 ppc-tools-skill 确认）

## 核心原则

> **旧版不动，新版另存。** 每次修改都是一次版本快照，用户可回退到任意历史版本。

---

## 执行流程

### Step 1：版本快照——冻结当前版本

在修改之前，对当前工作目录做快照：

```bash
# 在工作目录下创建版本目录
mkdir -p versions/v001
cp 排版大纲.md versions/v001/
cp main.tex versions/v001/
cp main.pdf versions/v001/
cp references.bib versions/v001/ 2>/dev/null
# 复制图片目录
cp -r figures/ versions/v001/figures/ 2>/dev/null
```

**版本编号规则**：
- 首次修改 → `v001`（论文初稿）
- 第二次修改 → `v002`
- 以此类推，永不覆盖旧版本

同时生成版本日志 `versions/CHANGELOG.md`：

```markdown
## 版本历史

| 版本 | 时间 | 修改类型 | 修改摘要 |
|------|------|----------|----------|
| v001 | YYYY-MM-DD HH:MM | 初稿 | ppc-edit-skill 生成的首版论文 |
| v002 | YYYY-MM-DD HH:MM | 用户修改 | 修改摘要... |
```

### Step 2：接收用户修改意见

向用户确认修改内容。如果用户描述模糊，追问澄清：

```
请确认你的修改要求：
1. 改什么？（内容 / 格式 / 结构 / 参考文献 / 图表）
2. 怎么改？（具体描述）
3. 改哪里？（全文 / 某个章节 / 某个图表）
```

**询问模板**（如果用户只是说"改一下格式"这种模糊描述）：

> "请具体说明需要修改的内容。例如：
> - 格式类：'正文字号改为小四'、'行距改为 1.5 倍'、'页边距改为上下 2.5cm 左右 3cm'
> - 内容类：'第二章增加一段关于 XXX 的讨论'、'图 3 换为最新实验结果'
> - 结构类：'把第三章和第四章顺序对调'、'增加一个附录'
> - 引用类：'参考文献改用 APA 7'、'增加 \cite{xxx} 这条引用'"

### Step 3：复制排版大纲

```bash
cp 排版大纲.md 排版大纲_v002.md   # 版本号对应当前修改轮次
```

**在副本中修改。原始 `排版大纲.md` 保持不动。**

### Step 4：在副本大纲中严格按用户要求修改

逐条处理用户修改意见，按类型分别操作：

#### 格式类修改

直接更新排版大纲对应章节：

```
用户："正文字号改为小四"
  → 排版大纲 → 第二章（字体体系）→ "中文正文：宋体 小四" ✅

用户："页边距改为上下 2.5cm 左右 3cm"
  → 排版大纲 → 第一章（页面基础）→ 更新 \geometry{...} ✅
```

#### 内容类修改

在排版大纲的"内容与结构映射"章节中标注修改点：

```
用户："第二章增加一段关于 XXX 的讨论"
  → 排版大纲 → 第七章（内容与结构映射）→ 标注：
    [修改 v002] 2. 文献综述 → 新增 2.4 节 "XXX 讨论"，预计 500 字
```

#### 结构类修改

更新排版大纲的章节顺序和映射关系：

```
用户："把第三章和第四章顺序对调"
  → 排版大纲 → 第七章（内容与结构映射）→ 调整：
    3. [原第四章] 实验
    4. [原第三章] 方法
```

#### 引用类修改

更新排版大纲的参考文献章节：

```
用户："参考文献改用 APA 7"
  → 排版大纲 → 第六章（参考文献格式）→ 更新著录规则
  → 排版大纲 → 第十章（LaTeX 实现清单）→ biblatex 参数更新
```

#### 混合类修改

如果用户同时提了多条不同类型的修改，在大纲末尾追加一节：

```markdown
## 十一、v002 修改日志

| # | 用户要求 | 类型 | 影响章节 | 状态 |
|---|----------|------|----------|:---:|
| 1 | 正文字号改为小四 | 格式 | 二、七 | ✅ |
| 2 | 第二章增加 XXX 讨论 | 内容 | 七 | ✅ |
| 3 | 参考文献改用 APA 7 | 引用 | 六、十 | ✅ |
```

### Step 5：确认修改后的大纲

将修改后的排版大纲展示给用户确认：

> **"以下是修改后的排版大纲变更摘要（完整大纲见 `排版大纲_v002.md`）：**
>
> **格式变更**：X 项
> **内容变更**：Y 项
> **结构变更**：Z 项
> **引用变更**：W 项
>
> **确认无误后我将重新编译论文。是否继续？"**

用户确认前不执行编译。

### Step 6：按 ppc-edit-skill 步骤重新编译

以 `排版大纲_v002.md` 为唯一权威，严格按照 ppc-edit-skill 的 11 步流程重新生成论文：

```
第一轮（技术编译）：
  ├── 修改 main.tex（基于新大纲）
  ├── 首次编译 → xelatex main.tex
  ├── 消错误 → 逐条修复
  ├── 参考文献 → biber main
  └── 交叉引用 → xelatex ×2

第二轮（质量打磨）：
  ├── 图表优化
  ├── 版式精调
  ├── 语言校对（chktex + textidote + 人工）
  └── 自查清单

第三轮（交付）：
  ├── 最终编译 → latexmk -C && latexmk -xelatex
  ├── 格式导出（如需要）
  └── 交付校验
```

编译完成后，新论文输出为：

```
main_v002.tex       # 新版源文件
main_v002.pdf       # 新版 PDF
references.bib      # 参考文献（如未变则共用）
```

### Step 7：更新版本日志

```markdown
| v002 | YYYY-MM-DD HH:MM | 用户修改 | 字号改小四；第二章新增XXX讨论；参考文献改APA 7 |
```

## 文件结构约定

修改后工作目录的文件布局：

```
工作目录/
│
├── 排版大纲.md               ← 原始大纲（只读，不改）
├── 排版大纲_v002.md           ← 本轮修改后的大纲副本
├── 排版大纲_v001.md           ← 上一轮的大纲副本（如有）
│
├── main.tex                  ← 最新版源文件
├── main.pdf                  ← 最新版 PDF
│
├── versions/
│   ├── CHANGELOG.md          ← 版本日志
│   ├── v001/                 ← 论文初稿快照
│   │   ├── 排版大纲.md
│   │   ├── main.tex
│   │   └── main.pdf
│   └── v002/                 ← 本轮修改前快照（当前 main.* 的上一版）
│       ├── 排版大纲.md
│       ├── main.tex
│       └── main.pdf
│
├── references.bib
└── figures/
```

---

## 多轮修改

```
v001 (初稿) → v002 (第一次修改) → v003 (第二次修改) → ...
                                    ↑
                              每次从当前最新大纲复制 → 修改副本 → 重新编译
                              versions/ 下保留全部历史版本
```

每轮修改前，将当前最新版快照到 `versions/v00N/`，然后基于 `排版大纲_v00N.md` 修改。

## 回退机制

如果用户说"不满意，回到上一版"：

```bash
# 回退到 v001（论文初稿）
cp versions/v001/main.tex main.tex
cp versions/v001/排版大纲.md 排版大纲.md
# 重新编译
xelatex main.tex && biber main && xelatex main.tex && xelatex main.tex
```

---

## 关键规则

1. **旧文件绝不动** — 原始排版大纲和旧论文只读，所有修改在副本上做
2. **严格遵循用户要求** — 不擅自发挥、不"我觉得这样更好"
3. **每改必确认** — 修改大纲后、编译前，必须让用户确认变更摘要
4. **版本可追溯** — versions/ 目录 + CHANGELOG.md 保证任何版本都能恢复
5. **编译步骤不跳过** — 即使是"只改一个字"，也要走完三阶段编译流程（确保交叉引用不受影响）
6. **工具路径** — 以 `ppc-tools-skill` 检测到的路径为准
